# FSD 전면 리팩토링 설계 문서

## 1. 목적

이 리팩토링의 목적은 단순한 폴더 정리가 아니다.  
프로젝트가 커질수록 다음 문제가 반복될 가능성이 높기 때문에, 이를 구조적으로 막기 위한 기준을 먼저 정의한다.

- AI와 사람이 함께 읽고 수정하기 쉬운 구조로 정리한다.
- 파일 간 참조가 과도하게 얽히는 현상을 방지한다.
- 전투, 텍스트, 연출, 도메인 규칙이 한 파일에 섞이는 문제를 줄인다.
- 텍스트 중심 게임이라는 프로젝트 특성에 맞춰, 텍스트 자산을 탐색하고 수정하기 쉬운 형태로 재구성한다.

즉, 이번 전환의 핵심은 `폴더 이동`이 아니라 `책임 분리`, `의존 방향 고정`, `텍스트 관리 규칙 수립`이다.

## 2. 이번 리팩토링의 범위

이번 리팩토링은 다음 범위를 포함한다.

- `src` 디렉토리를 FSD 기반 구조로 재편한다.
- 앱 엔트리, 페이지, 위젯, 기능, 엔티티, 공용 계층의 역할을 나눈다.
- 텍스트와 현지화 데이터를 별도 규칙 아래 재배치한다.
- 현재 중복되거나 역할이 겹치는 i18n 관련 파일을 통합한다.
- 대형 파일을 기능 단위 또는 책임 단위로 분해한다.
- 향후 제작, 탐험, 사전 시스템이 추가되어도 구조가 무너지지 않도록 기반을 만든다.

다음 작업은 이번 범위에 포함하지 않는다.

- 전투 밸런스 조정
- 시스템 기획 변경
- 연출 리디자인
- UI 스타일 리뉴얼
- 신규 기능 추가 자체

## 3. 현재 구조의 문제 요약

현재 구조는 빠르게 프로토타입을 만들기에는 적합하지만, 확장성과 유지보수성 측면에서 한계가 분명하다.

- 루트 `src` 아래에 기능과 도메인이 평평하게 나열되어 있다.
- `App`, `BattleScene`, `BattleCombat`에 상태, 텍스트, 연출, 규칙이 함께 섞여 있다.
- 텍스트가 컴포넌트 내부 상수, `language.ts`, `i18n.ts`, 도메인 데이터 파일에 분산되어 있다.
- 도메인 규칙 파일 안에 UI 표기용 문자열과 데이터가 함께 들어 있다.
- 일부 파일은 현재 사용되지 않거나, 역할이 겹칠 가능성이 있다.

이 상태에서 기능이 늘어나면 다음과 같은 문제가 커진다.

- 새로운 기능을 넣을 때 어디에 작성해야 할지 기준이 모호해진다.
- 한 파일을 수정할 때 다른 시스템에 영향을 줄 가능성이 높아진다.
- AI가 수정할 때 문맥 범위가 너무 넓어져 정확도가 떨어진다.
- 텍스트 수정만 하더라도 로직 파일을 열어야 하는 경우가 많아진다.

## 4. 목표 구조

프로젝트는 표준 FSD를 그대로 복제하지 않고, 현재 게임 구조에 맞게 약간 조정한 형태로 정리한다.

```text
src/
  app/
    providers/
    styles/
    App.tsx
    main.tsx

  pages/
    campfire/
    battle/
    post-battle-event/

  widgets/
    battle-stage/
    battle-log/
    resource-panel/
    encounter-scene/

  features/
    language-switch/
    bonfire-start/
    battle-command-input/
    potion-use/
    equipment-choice/

  entities/
    combat/
    player/
    monster/
    spell/
    equipment/
    locale/

  shared/
    lib/
    ui/
    config/
    assets/

  content/
    text/
      app/
      battle/
      event/
    glossary/
      spells/
      monsters/
    catalog/
      equipment/
      recipes/
```

## 5. 계층별 역할

### `app`

앱의 실행 진입점과 전역 설정만 둔다.

- 루트 앱 셸
- 전역 provider
- 전역 스타일
- 라우팅 또는 최상위 phase 연결

여기에는 게임 규칙, 텍스트 본문, 전투 판정 로직을 두지 않는다.

### `pages`

하나의 장면 또는 화면 단위 조합을 담당한다.

- 캠프파이어 시작 장면
- 전투 장면
- 전투 후 이벤트 장면

`pages`는 여러 `widgets`와 `features`를 조합하지만, 세부 계산 로직을 직접 들고 있지 않도록 한다.

### `widgets`

하나의 화면에서 덩어리로 재사용되거나, 복합 UI를 구성하는 단위를 둔다.

- 전투 무대 전체
- 전투 로그 패널
- 자원 패널
- 조우 연출 블록

`BattleCombat.tsx`처럼 지나치게 큰 파일은 이 계층에서 다시 내부 모듈로 나누는 방향을 기본으로 한다.

### `features`

사용자 행동 중심의 기능을 둔다.

- 언어 전환
- 전투 입력 처리
- 포션 사용
- 장비 선택
- 제작 시도

핵심 기준은 "사용자가 무엇을 한다"이다.  
도메인 데이터 자체는 여기 두지 않는다.

### `entities`

게임의 핵심 개체와 그에 대한 규칙, 타입, 계산 함수를 둔다.

- 플레이어 스탯
- 전투 액션/타깃/판정
- 몬스터 정의
- 주문 정의
- 장비 정의
- 언어/로케일 모델

예를 들어 기존 `battleTypes.ts`에 섞여 있던 플레이어 계산식, 장비 데이터, 주문 정의 등은 이 계층으로 분리하는 것이 맞다.

### `shared`

도메인에 종속되지 않는 공용 자원을 둔다.

- 공용 유틸 함수
- 공용 UI 컴포넌트
- 전역 설정값
- 정적 자산 접근 유틸

여기에는 특정 몬스터, 특정 장비, 특정 장면의 문구를 두지 않는다.

### `content`

이 프로젝트에서 새로 강조하는 계층이다.  
텍스트 중심 게임 특성상, 플레이어가 읽는 내용과 데이터성 컨텐츠를 기능 코드와 분리해 관리한다.

- 장면별 서사 텍스트
- 전투 문구
- 이벤트 문구
- 도감/사전용 텍스트
- 장비/몬스터/주문의 이름과 설명
- 향후 제작 레시피 설명 및 탐험 설명문

`content`는 로직을 가지지 않는 `typed data layer`로 유지한다.

## 6. 텍스트 자산 관리 원칙

텍스트를 한 폴더에 모으고 싶다는 방향 자체는 타당하다.  
다만 모든 문구를 하나의 거대한 전역 파일에 몰아넣는 방식은 피한다.

### 채택 방안

- 텍스트는 `src/content` 계층으로 분리한다.
- 텍스트는 `도메인 또는 장면 단위`로 나눈다.
- 로직 파일 내부에 긴 서사 문구를 직접 작성하지 않는다.
- 텍스트는 가능하면 TypeScript 객체로 보관한다.

### 권장 예시

- `src/content/text/app/campfire.ts`
- `src/content/text/battle/scene.ts`
- `src/content/text/battle/log.ts`
- `src/content/text/event/postBattle.ts`
- `src/content/glossary/spells/spellNames.ts`
- `src/content/catalog/equipment/equipmentText.ts`

### 이렇게 하지 않는 이유

다음 방식은 피한다.

- `src/text.ts` 하나에 모든 문구를 몰아넣는 방식
- 도메인 규칙 파일에 사용자 노출 텍스트를 계속 추가하는 방식
- 컴포넌트 내부 상수로 서사 텍스트를 계속 누적하는 방식

이유는 다음과 같다.

- 전역 텍스트 파일은 빠르게 비대해진다.
- 기능 문맥이 끊겨 수정 시 실수가 늘어난다.
- 번역, 참조, 데이터 검증이 어려워진다.
- AI가 필요한 텍스트 묶음만 좁게 읽기 어려워진다.

## 7. 현지화 구조 원칙

현재 언어 관련 구조는 중복 가능성이 있으므로 하나의 소스로 통합한다.

원칙은 다음과 같다.

- 로케일 타입은 하나만 유지한다.
- 초기 로케일 결정 함수도 한 곳에만 둔다.
- `pickText` 성격의 헬퍼도 하나의 locale 모델 아래 정리한다.
- 번역 데이터는 `content` 계층에 두고, 이를 읽는 유틸만 `entities/locale` 또는 `shared/i18n` 성격의 위치에 둔다.

권장 방향은 아래 둘 중 하나다.

1. `entities/locale`
   - 언어 타입
   - locale selector
   - 번역 키 타입
   - locale 헬퍼

2. `content/text`
   - 실제 번역 데이터

즉, 언어 로직과 언어 데이터를 분리한다.

## 8. 의존 방향 규칙

이번 리팩토링의 핵심은 폴더 구조보다 의존 방향이다.

기본 원칙:

- `app`은 아래 모든 계층을 사용할 수 있다.
- `pages`는 `widgets`, `features`, `entities`, `shared`, `content`를 사용할 수 있다.
- `widgets`는 `features`, `entities`, `shared`, `content`를 사용할 수 있다.
- `features`는 `entities`, `shared`, `content`를 사용할 수 있다.
- `entities`는 `shared`, `content`를 사용할 수 있다.
- `shared`는 다른 상위 계층을 참조하지 않는다.
- `content`는 어떤 계층도 참조하지 않는다.

추가 원칙:

- 상위 계층이 하위 계층을 참조하는 것은 허용한다.
- 하위 계층이 상위 계층을 참조하는 것은 금지한다.
- slice 간 deep import를 피하고 `index.ts`를 통한 public API를 우선한다.
- 다른 slice 내부 구현 파일을 직접 참조하지 않는다.

## 9. Public API 원칙

각 slice는 외부에서 접근할 최소 표면만 노출한다.

예시:

- `entities/spell/index.ts`
- `features/language-switch/index.ts`
- `widgets/battle-stage/index.ts`

외부에서 사용할 때는 내부 파일 경로가 아니라 `index.ts`를 통해 import하는 것을 기본 규칙으로 둔다.

이 원칙이 필요한 이유는 다음과 같다.

- 파일 이동 시 영향 범위를 줄일 수 있다.
- 내부 구현 변경이 외부 의존에 바로 새지 않는다.
- AI가 "무엇이 공식 진입점인지" 판단하기 쉬워진다.

## 10. 대형 파일 분해 원칙

이번 리팩토링에서는 특히 다음 파일을 우선 분해 대상으로 본다.

- `App.tsx`
- `BattleScene.tsx`
- `BattleCombat.tsx`
- 기존 `battleTypes.ts`에서 분리된 전투 도메인 묶음

분해 기준은 다음과 같다.

- 장면 조합과 상태 전환은 `pages` 또는 `widgets`로 이동한다.
- 도메인 계산식은 `entities`로 이동한다.
- 텍스트 상수는 `content`로 이동한다.
- 순수 렌더링 유틸은 `shared` 또는 widget 내부 `lib`로 이동한다.
- 특정 사용자 행동은 `features`로 이동한다.

특히 `BattleCombat.tsx`는 한 번에 완전히 쪼개기보다 아래 순서로 단계 분해한다.

1. 입력 관련
2. 자원 UI 관련
3. 연출 오버레이 관련
4. 캔버스 렌더링 보조 로직 관련
5. 텍스트/프롬프트 관련

## 11. 마이그레이션 단계

전면 리팩토링은 반드시 단계적으로 진행한다.

### 1단계. 구조 골격과 별칭

- `src` 하위 새 디렉토리 생성
- tsconfig 및 Vite alias 설정
- import 규칙 기준 확정
- README에 새 구조 개요 추가

### 2단계. locale 및 text 통합

- `language.ts`, `i18n.ts` 중복 구조 정리
- 로케일 모델 통합
- 플레이어 노출 텍스트를 `content`로 이동
- 텍스트 참조 경로를 새 구조로 교체

### 3단계. 도메인 분리

- 기존 `battleTypes.ts`를 역할별로 분해
- player, monster, spell, equipment 규칙 이동
- 장비 텍스트와 규칙 데이터 분리 여부 확정

### 4단계. 전투 화면 분해

- `BattleScene.tsx`와 `BattleCombat.tsx` 분해
- 전투 stage, 로그, 자원 패널, 입력 기능 분리
- 순수 시각 효과와 장면 조합 책임 분리

### 5단계. 정리 및 검증

- 미사용 파일 정리
- 참조 경로 정리
- 문서 업데이트
- lint/build 검증

## 12. 성공 기준

리팩토링 완료 기준은 다음과 같다.

- 새 기능을 추가할 때 위치 선택 기준이 명확하다.
- 플레이어 노출 텍스트를 한눈에 찾을 수 있다.
- 텍스트 수정만을 위해 거대한 로직 파일을 열지 않아도 된다.
- 대형 파일이 역할 기준으로 분리되어 있다.
- import 경로가 임의로 꼬이지 않고, public API 중심으로 유지된다.
- `npm run lint`, `npm run build`가 통과한다.

## 13. 리스크와 주의사항

가장 큰 리스크는 "이름만 FSD"가 되는 것이다.

다음 상황은 반드시 피한다.

- 파일만 옮기고 책임은 그대로 두는 경우
- `content`를 만들었지만 다시 컴포넌트 내부에 텍스트를 계속 추가하는 경우
- `index.ts` 없이 deep import를 계속 사용하는 경우
- 사용하지 않는 파일을 새 구조로 그대로 들고 가는 경우
- 한 번의 대형 PR로 모든 파일을 이동하는 경우

이번 리팩토링은 `큰 이동`보다 `명확한 규칙 확정 + 작은 단위 이행`이 더 중요하다.

## 14. 결론

이 프로젝트는 일반적인 CRUD 앱보다 텍스트와 연출 비중이 높기 때문에, 표준 FSD만으로는 부족하다.  
따라서 이번 개편은 `FSD + content 계층`을 기본 원칙으로 삼는다.

핵심 요약:

- 구조는 FSD 기반으로 재편한다.
- 텍스트는 `content` 계층으로 분리한다.
- 로케일 로직과 번역 데이터는 분리한다.
- 대형 파일은 책임 기준으로 단계 분해한다.
- public API와 의존 방향을 문서화하고 지킨다.

이 문서는 실제 파일 이동 전에 팀과 작업 기준을 맞추기 위한 설계 기준 문서로 사용한다.
