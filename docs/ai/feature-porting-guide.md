# 리팩토링 이후 기능 이식 상세 가이드

## 1. 문서 목적

이 문서는 삭제된 `feat/#4` 계열의 예전 구조에서 시작된 기능을  
현재 구조로 옮길 때 참고하는 상세 가이드다.

핵심 목적은 다음 세 가지다.

- 기존 기능을 현재 구조에 맞게 안전하게 이식한다.
- 삭제된 파일 구조나 평면 `src` 구조를 다시 되살리지 않는다.
- AI가 "어디를 먼저 읽고, 무엇을 어떤 계층에 놓아야 하는지" 빠르게 판단할 수 있게 한다.

## 2. 이 문서를 읽어야 하는 작업

다음과 같은 경우 이 문서를 반드시 읽는다.

- 리팩토링 이전 브랜치에서 개발된 기능을 현재 구조로 이식할 때
- 다른 팀원의 기능 브랜치를 현재 구조 기준으로 다시 반영할 때
- `git cherry-pick` 대신 책임 단위 재이식이 필요한 작업일 때

## 3. 먼저 읽을 문서

기능 이식 작업을 시작하기 전에 아래 문서를 순서대로 읽는다.

1. `AGENTS.md`
2. `docs/ai/feature-porting-guide.md`
3. 기능별 세부 설계 문서가 있으면 그 문서

원칙:

- 예전 리팩토링용 문서는 기본 참고 대상이 아니다.
- 과거 구조 문서는 사용자가 명시적으로 요구할 때만 본다.

## 4. 디렉토리 읽는 순서

저장소는 아래 순서로 읽는다.

1. `src/app`
   - 앱 진입점과 전역 셸만 본다.
   - 기능 구현 세부 사항을 찾는 용도가 아니다.
2. `src/pages`
   - 어떤 화면이 어떤 widget / feature를 조합하는지 본다.
   - 기능이 "어느 장면에 연결되는지" 먼저 파악한다.
3. `src/widgets`
   - 화면의 큰 UI 블록을 본다.
   - 전투 관련 기능이라면 우선 `widgets/battle-stage`, `widgets/battle-log`, `widgets/resource-panel`을 확인한다.
4. `src/features`
   - 사용자의 행동 단위를 본다.
   - 입력, 선택, 사용, 전환 같은 상호작용은 먼저 여기서 위치를 찾는다.
5. `src/entities`
   - 도메인 규칙, 타입, 계산 로직을 본다.
   - 기능의 핵심 판정이나 데이터 구조가 바뀌는지 여기서 확인한다.
6. `src/shared`
   - 완전히 공용인 유틸과 UI만 본다.
   - 특정 기능 구현을 여기로 밀어 넣지 않는다.
7. `src/content`
   - 플레이어 노출 텍스트, 설명, 카탈로그 데이터를 본다.
   - 새 텍스트가 필요하면 가장 먼저 여기 배치할 위치를 정한다.

## 5. 전투 기능 이식 시 읽는 순서

전투 관련 기능이라면 아래 순서로 좁혀 읽는다.

1. `src/pages/battle/ui/BattlePage.tsx`
   - 장면 조합과 전투 진입/종료 흐름을 파악한다.
2. `src/widgets/battle-stage/ui/BattleStage.tsx`
   - 실제 화면 조합 지점과 DOM ref 연결 지점을 본다.
3. `src/widgets/battle-stage/model/*`
   - canvas loop, 피드백, 상태 전이, 전투 플로우를 책임 단위로 확인한다.
4. `src/features/battle-command-input/*`, `src/features/potion-use/*`
   - 사용자 입력 기반 기능인지 판단한다.
5. `src/entities/combat/*`, `src/entities/player/*`, `src/entities/monster/*`, `src/entities/spell/*`
   - 판정 규칙과 도메인 타입이 바뀌는지 확인한다.
6. `src/content/text/battle/*`
   - 로그, 버튼, 안내 문구 등 노출 텍스트를 확인한다.

## 6. 기능 이식 절차

기능 이식은 "옛 파일을 복사"하는 작업이 아니라 "기존 변경 의도를 현재 구조로 재배치"하는 작업이다.

권장 순서는 다음과 같다.

1. 원본 범위를 확정한다.
   - 현재 구조 기준 브랜치, 원본 기능 브랜치, 변경 의도 추출 기준 ref, 핵심 커밋 범위를 먼저 정한다.
   - 예시:

```powershell
git log --oneline c8138f4..origin/feat/#5
git diff --name-only c8138f4..origin/feat/#5
```

현재 사례 해석:

- 현재 구조 기준 브랜치: `feat/#6`
- 원본 기능 브랜치: `origin/feat/#5`
- `origin/feat/#5`가 담고 있는 기능: `voca lexicon and word prompt polish`
- 변경 의도 추출 기준: `c8138f4` (`origin/feat/#4`, `feat/#4`, `main`과 공통)
- `c8138f4`는 `feat/#4`의 마지막 커밋이며, 메시지는 `chore: github page 배포작업`

해석 규칙:

- `feat/#4`의 마지막 상태 자체는 기능 diff 대상이 아니다.
- `c8138f4..origin/feat/#5` 범위에서 새로 생긴 변경만 `voca lexicon and word prompt polish`의 실제 차이로 본다.
- 즉, 새 세션에서는 먼저 이 범위의 commit / file diff를 읽고 나서 책임 매핑을 시작한다.

2. 변경 의도를 파일이 아니라 책임 기준으로 분해한다.
   - UI 표현 변경인지
   - 사용자 상호작용 변경인지
   - 도메인 규칙 변경인지
   - 텍스트/데이터 추가인지
   - 공용 유틸 추가인지

3. 현재 구조의 목표 위치를 먼저 정한다.
   - 장면 조합이면 `pages`
   - 큰 UI 블록이면 `widgets`
   - 사용자 행동이면 `features`
   - 규칙/타입이면 `entities`
   - 완전 공용이면 `shared`
   - 플레이어 노출 텍스트면 `content`

4. 기능을 계층 순서대로 이식한다.
   - `entities` / `content` / `shared`부터 이식
   - 그다음 `features` / `widgets`
   - 마지막에 `pages` / `app` 연결

5. 기존 리팩토링 경계를 유지한 채 배선한다.
   - 이미 분리된 model 훅이 있다면 그 내부 책임에 맞게 넣는다.
   - 다시 `BattleStage.tsx` 같은 조합 컴포넌트에 로직을 몰아넣지 않는다.

6. 문서와 검증을 같이 마무리한다.
   - 구조 책임이 바뀌면 문서를 갱신한다.
   - `npm run lint`, `npm run build`를 통과시킨다.

## 7. 기능 이식 시 계층 배치 기준

기능 이식 중 자주 헷갈리는 기준은 아래를 따른다.

- 기존 브랜치에서 `BattleScene.tsx`나 `BattleCombat.tsx`를 수정했더라도, 현재는 그대로 대응되는 단일 파일이 없다고 가정한다.
- 기존 브랜치에서 입력/버튼/드래그/선택 로직을 넣었다면, 현재는 먼저 `features/*`에 갈 수 있는지 본다.
- 기존 브랜치에서 전투 연출을 넣었다면, 현재는 `widgets/battle-stage/model/*` 또는 `widgets/battle-stage/lib/*`을 먼저 본다.
- 기존 브랜치에서 로그 문구나 버튼 텍스트를 넣었다면, 현재는 `content/text/*`로 먼저 보낸다.
- 기존 브랜치에서 타입이나 판정 로직을 추가했다면, 현재는 `entities/*`에 둔다.

## 8. 금지 사항

기능 이식 작업에서 아래 패턴은 금지한다.

- 과거 브랜치의 파일을 통째로 복사해 현재 구조 위에 덮어쓰는 것
- 삭제된 루트 `src` 평면 파일을 다시 만드는 것
- `widgets`, `features`, `entities`의 다른 slice 내부 파일을 deep import 하는 것
- 플레이어 노출 텍스트를 다시 컴포넌트 내부 상수로 넣는 것
- `content` 데이터를 다시 집계용 거대 `index.ts`로 모으는 것
- 기존 기능을 이식하면서 무관한 UI 리디자인이나 밸런스 변경을 같이 섞는 것

## 10. 완료 체크리스트

아래 항목을 모두 만족하면 기능 이식 작업 완료로 본다.

- 원본 기능의 핵심 동작이 현재 구조에서 재현된다.
- 삭제된 예전 파일 경로를 다시 만들지 않았다.
- `content`와 도메인 규칙이 다시 섞이지 않았다.
- 새 import가 FSD 계층 규칙을 어기지 않는다.
- 필요한 문서를 갱신했다.
- `npm run lint`
- `npm run build`
