# 웹 게임(TypeScript) 이슈 기반 개발 시작 가이드

## 1) 브랜치/이슈 규칙 먼저 고정

팀 규칙을 아래처럼 고정하면 충돌이 크게 줄어듭니다.

- 이슈 제목 규칙: `#<번호> [타입] : 작업 설명`
  - 예: `#3 [feat] : 전투 기능 개발`
- 브랜치 규칙: `<type>/#<issue-number>`
  - 예: `feat/#3`, `fix/#7`, `chore/#12`
- 커밋 메시지 규칙(권장): `<type>(#<번호>): <변경 요약>`
  - 예: `feat(#3): 전투 턴 계산 로직 추가`
- PR 제목 규칙(권장): `[<type>] #<번호> <작업 요약>`

> 참고: Git 호스팅/CLI에 따라 `#`가 포함된 브랜치명 사용이 불편할 수 있어, 실제 운영에서는 `feat/3` 형식도 많이 씁니다.

---

## 2) 초기 기술 스택 제안 (웹 게임 + TypeScript)

빠르게 시작하려면 아래 조합을 추천합니다.

- 런타임/빌드: `Vite + TypeScript`
- 렌더링(선택):
  - UI 중심 게임: `React + Canvas`
  - 2D 게임 중심: `Phaser + TypeScript`
- 코드 품질: `ESLint + Prettier`
- 테스트:
  - 로직: `Vitest`
  - E2E(선택): `Playwright`
- 상태/데이터(선택): `Zustand` 또는 순수 상태 모듈

---

## 3) 처음 만들 이슈 백로그 (추천 순서)

1. `chore`: 프로젝트 초기화 (Vite/TS/ESLint/Prettier)
2. `chore`: 폴더 구조 확정 (`src/core`, `src/features`, `src/shared`)
3. `feat`: 게임 루프(타이머, 업데이트, 렌더)
4. `feat`: 입력 시스템(키보드/마우스)
5. `feat`: 씬 전환(타이틀/게임/결과)
6. `feat`: 전투 기본 규칙(턴, HP, 데미지)
7. `feat`: 저장/불러오기(LocalStorage)
8. `test`: 핵심 전투 로직 단위 테스트
9. `ci`: lint/test 자동화(GitHub Actions)

핵심은 **기능 이슈를 작게 쪼개는 것**입니다. (1 PR = 1 이슈)

---

## 4) Codex 활용법 (실무형)

Codex는 "기획 정리 → 코드 생성 → 리팩터링 → 테스트 보강 → PR 문서화" 흐름에서 가장 효율적입니다.

### A. 이슈 정의 단계에서

Codex에게 이렇게 요청하세요:

- "이 이슈를 0.5~1일 단위 서브태스크로 쪼개줘"
- "완료 조건(DoD)과 테스트 케이스를 같이 만들어줘"
- "리스크/선행 의존성 체크리스트를 만들어줘"

### B. 구현 단계에서

- "`src/features/battle`에 타입 중심으로 설계해줘"
- "상태 전이 다이어그램을 코드 구조로 옮겨줘"
- "랜덤 요소를 테스트 가능하게 인터페이스 분리해줘"

### C. 리뷰 단계에서

- "이 PR에서 버그 가능성 높은 지점 5개 찾아줘"
- "성능/가독성 개선 포인트를 우선순위로 정리해줘"
- "테스트 누락 케이스를 추가해줘"

### D. 자동화/운영 단계에서

- "커밋 메시지/PR 템플릿을 팀 규칙에 맞춰 생성해줘"
- "브랜치 전략 문서와 온보딩 문서를 갱신해줘"

---

## 5) Codex 프롬프트 템플릿 (복붙용)

### 이슈 시작 프롬프트

```text
이슈: #3 [feat] : 전투 기능 개발
브랜치: feat/#3
목표: 턴 기반 전투 MVP 완성
제약: TypeScript strict, 테스트 필수, UI보다 도메인 로직 우선
요청:
1) 작업을 파일 단위로 설계
2) 구현 순서 제안
3) 단위 테스트 케이스 먼저 작성
4) 커밋 단위까지 쪼개서 제안
```

### PR 직전 프롬프트

```text
다음 변경사항을 리뷰해줘:
- 버그 가능성
- 타입 안정성
- 테스트 누락
- 리팩터링 우선순위
그리고 PR 설명(문제/해결/테스트/리스크)을 작성해줘.
```

---

## 6) 팀 운영 팁

- 브랜치 생성 즉시 Draft PR을 열어 진행상황 공유
- "작업 중" 이슈에는 반드시 담당자/예상 완료일 기록
- PR은 300~500줄 내로 작게 유지
- 게임 밸런스 수치(데미지, 쿨다운)는 상수 파일로 분리
- 랜덤 로직은 시드 주입 가능하게 만들어 재현성 확보

---

## 7) 바로 시작하는 오늘의 체크리스트

- [ ] `chore/#1` 브랜치에서 프로젝트 초기화
- [ ] 린트/포맷/테스트 스크립트 추가
- [ ] CI에서 `lint + test` 자동 실행
- [ ] `feat/#2`에서 게임 루프 MVP 구현
- [ ] 전투 로직 이슈(`#3`)를 테스트 우선으로 진행

필요하면 다음 단계로, 이 저장소 기준으로 **초기 폴더 구조 + 기본 설정 파일(tsconfig/eslint/vitest)까지 바로 생성**할 수 있습니다.
