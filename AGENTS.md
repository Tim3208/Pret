# Pret 저장소 작업 가이드

## 목적

이 저장소에서 AI가 기능을 추가하거나 수정할 때 가장 먼저 지켜야 하는 기준은
지금의 파일 구조를 유지하는 것이다.

여기서 말하는 "지금의 파일 구조를 지킨다"의 정확한 의미는 다음과 같다.

- 현재 저장소에 이미 자리잡은 FSD 기반 디렉토리 구조를 기본으로 본다.
- AI가 작업하기 쉽게 책임을 분리하되, 사람이 읽어도 흐름을 따라갈 수 있게 유지한다.
- 과도하게 잘게 쪼개거나, 반대로 책임이 다른 코드를 한 파일에 다시 뭉치지 않는다.
- 새 기능은 기존 구조에 맞춰 배치하고, 구조 자체를 임의로 다시 설계하지 않는다.
- 구조 재설계가 필요하다고 판단되면 먼저 사용자에게 변경 이유와 범위를 제안하고 승인을 받는다. 승인 없이 구조를 변경하지 않으며, 작업 후에는 관련 문서를 반드시 갱신한다.

## 최우선 원칙

- 현재 `src` 구조를 기준으로 작업한다.
- 삭제된 예전 평면 `src` 구조를 복원하지 않는다.
- 기능 추가를 이유로 FSD 경계를 무너뜨리지 않는다.
- AI 친화적인 분리는 가능하지만, 인간이 읽기 어려운 과분할은 피한다.
- 한 파일의 책임은 명확해야 하고, 파일 이름만 보고도 역할을 짐작할 수 있어야 한다.

## 계층 배치 기준

- `app`: 앱 진입점, 전역 셸, 전역 provider, 전역 스타일
- `pages`: 화면 단위 조합
- `widgets`: 화면 안의 큰 장면 UI 블록
- `features`: 사용자 행동, 입력, 선택, 사용
- `entities`: 타입, 규칙, 계산, 판정, 도메인 모델
- `shared`: 완전 공용 유틸과 공용 UI
- `content`: 플레이어 노출 텍스트, 설명, 로그, 카탈로그 데이터

판단 규칙:

- 화면 조합이면 `pages`
- 큰 UI 블록이면 `widgets`
- 사용자 상호작용이면 `features`
- 수치 계산, 판정, 타입이면 `entities`
- 여러 기능이 함께 쓰는 범용 코드면 `shared`
- 플레이어가 읽는 문장, 설명, 로그, 사전 데이터면 `content`

## 구조 유지 규칙

- 다른 slice 내부 파일로 deep import 하지 않는다.
- 가능하면 각 slice의 public API인 `index.ts`를 통해 접근한다.
- 특정 기능 전용 코드를 `shared`로 밀어 넣지 않는다.
- 플레이어 노출 텍스트를 컴포넌트 내부 상수에 직접 누적하지 않는다.
- 로직 파일 안에 UI 문구를 쌓아두지 않는다.
- 기존 구조에 맞는 위치가 있는데 새 중간 계층을 임의로 만들지 않는다.

## 구현 스타일

- 새 기능은 먼저 현재 구조 안에서 어느 계층 책임인지 확정한 뒤 구현한다.
- 기존 패턴이 있으면 그 패턴을 따른다.
- 무관한 리팩토링, 리디자인, 밸런스 변경을 기능 작업에 섞지 않는다.
- 이름은 역할이 드러나게 짓고, 축약어 남용을 피한다.
- AI가 보기 쉬운 코드도 중요하지만, 사람이 처음 읽어도 목적을 이해할 수 있어야 한다.

## Pretext 연출 원칙

- 플레이어가 읽는 핵심 연출 문장, 전투 피드백, 정비 결과, 도감/의식형 UI 텍스트는 가능한 Pretext 기반 줄바꿈과 표시 연출을 우선 고려한다.
- Pretext는 단순 장식이 아니라 문장이 화면 안에서 자연스럽게 드러나고, 게임의 몰입감을 높이는 방향으로 사용한다.
- 짧은 버튼 라벨, 접근성 라벨, 단순 숫자 상태처럼 Pretext 이점이 거의 없는 문구에는 무리하게 적용하지 않는다.
- Pretext 유틸이나 패널이 여러 계층에서 재사용된다면 `shared`에 두고, 특정 장면 전용 연출이면 해당 `widgets` 또는 `features` 안에 둔다.

## 함수와 주석 규칙

- 새로운 함수를 추가할 때는 알아보기 쉽게 한글 JSDoc 주석을 작성한다.
- 함수 주석에는 최소한 "무엇을 하는지"가 드러나야 한다.
- 입력이나 반환이 헷갈리면 `@param`, `@returns`를 함께 적는다.
- 변수가 복잡하거나 문맥이 없으면 변수에도 짧은 한글 주석을 달 수 있다.
- 다만 자명한 코드에 불필요한 주석을 늘리지 않는다.

예시:

```ts
/**
 * 플레이어 입력 문자열을 전투용 명령으로 정규화한다.
 */
function normalizeBattleCommand(input: string): string {
  return input.trim().toLowerCase();
}
```

## 문서 갱신 규칙

문서 갱신 규칙은 유지한다.

아래 경우에는 문서를 같이 갱신한다.

- 새 기능의 동작 규칙을 후속 작업자가 알아야 할 때
- 구조 배치 기준이 바뀌거나 새로 생겼을 때
- 특정 기능의 이식, 설계, 제약을 따로 남길 필요가 있을 때
- 저장소 전체에 적용되는 작업 원칙이 바뀔 때

기본 원칙:

- 저장소 전체 규칙은 `AGENTS.md`에 반영한다.
- Copilot 공통 지침을 바꿨다면 `.github/copilot-instructions.md`도 함께 갱신한다.
- 기능별 작업 메모나 설계 문서는 필요할 때 `docs/ai/` 아래에 새로 만든다.
- 문서가 오래되어 현재 코드와 어긋나면 코드 기준으로 다시 갱신한다.

## 문서 작성 기준

- 문서는 "현재 구조에서 어떻게 작업해야 하는지"를 설명해야 한다.
- 과거 구조 설명보다 현재 구조 기준 판단 규칙을 우선한다.
- 기능 문서는 작업 목적, 배치 이유, 검증 기준이 드러나게 쓴다.
- AI만 이해하는 표현보다, 사람이 읽어도 바로 판단 가능한 문장으로 쓴다.

## 금지 사항

- 삭제된 예전 파일 구조를 다시 만드는 것
- 예전 브랜치 파일을 통째로 복사해 덮어쓰는 것
- 다른 slice 내부 파일로 deep import 하는 것
- 플레이어 노출 텍스트를 로직 파일이나 컴포넌트에 직접 누적하는 것
- 특정 기능 전용 코드를 `shared`에 넣는 것
- 기능 작업과 무관한 대규모 리팩토링을 같이 하는 것

## 권장 작업 순서

1. `AGENTS.md`를 읽는다.
2. 관련 화면과 현재 구조를 먼저 읽는다.
3. 변경 의도를 책임 단위로 분해한다.
4. 어느 계층에 둘지 먼저 확정한다.
5. `entities` / `content` / `shared`부터 반영한다.
6. `features` / `widgets`를 반영한다.
7. `pages` / `app`을 연결한다.
8. 필요한 문서를 갱신한다.
9. 검증한다.

## 검증

작업 완료 전 아래를 기본 검증으로 본다.

- `npm run lint`
- `npm run build`

Windows PowerShell에서 `npm.ps1` 실행 정책 문제가 있으면 아래처럼 대체한다.

- `npm.cmd run lint`
- `npm.cmd run build`

## 추가 제안

아래 내용은 구조를 지키는 데 실제로 도움이 되므로 기본 규칙에 포함했다.

- public API인 `index.ts` 우선 사용
- `content`와 로직 분리
- `shared` 남용 금지
- 문서가 필요할 때 `docs/ai/`를 다시 만들어 관리
- 함수 추가 시 한글 JSDoc 필수
